# 前篇：整体准备清单（十段）

## 1. 产品与教学设计（非代码）

- **用户与场景**：学生 / 教师 / 管理员；典型课节长度、同时在线人数、是否直播、是否线下班级绑定。
- **课程结构模型**：章 → 节 → 知识点 → 例习题；**作业与章节的对应**（一章多份、死线、重做次数、是否防抄袭）。
- **Jupyter 在教学中的角色**：全站统一笔记本还是每章一份模板；是否允许 `pip install`、是否开放网络、单会话时长与 CPU/内存上限。
- **评价与成绩**：计分规则（仅教师打分 / 自动测试占比）、成绩单导出、**人工批改界面**的字段（评语、按题给分、迟交扣减等）。

## 2. 「零本地环境」约束下的架构决策（尽早定案）

- **应用形态**：纯浏览器；**Python 在服务端或托管环境**执行。需明确 **Jupyter 嵌入方式**（JupyterHub、托管 Notebook、自研 jupyter-server + 前端 Thebe/iframe 等；各方案对多租户与运维要求不同）。
- **多租户隔离**：一用户一容器 / 一会话一环境 / 共享池；**文件持久化**与作业草稿是否跨设备。
- **大模型数据流**：题面与代码是否回传模型；**是否必须境内部署与备案**；对话是否持久化、是否对教师开放督导。

**典型逻辑分层**（中间件名可随选型替换）：浏览器 → Web 应用（鉴权/课程）→ 业务 API → 数据存储；浏览器 → Notebook 执行服务；Web 应用 → LLM 网关；执行服务与数据存储可写回提交物与运行日志。

## 3. 账号与权限（含微信 / QQ）

- **身份模型**：`User`、`Role`（学生/教师/管理员）、**班级/课程成员**、作业可见范围。
- **凭据类**：密码策略、邮箱验证、**找回密码**、会话方式（Cookie/JWT）、是否上双因素。
- **微信 / QQ 登录**（国内常见路径）：各开放平台的应用类型；**已备案域名**、回调白名单、**公司主体与审核**材料。
- **统一身份**：建议内部主键 + OAuth 绑定表，避免多登录源冲突。
- **教师侧**：是否人工审核、邀请码、或后续与学校 SSO 对接。

## 4. 教学内容与作业子系统

- **内容管理**：富文本或 Markdown、代码块、附件；**版本与发布**（草稿/发布）；**编辑权限**（教师）。
- **作业系统**：题型（选择、简答、**Notebook 作业包**、自动测试）；**提交物**（`.ipynb`、日志、自动评分结果）；**批改队列**与通知（站内信/邮件/企业微信等，可选）。

## 5. Jupyter 嵌入与运行安全

- **运行时策略**：每 Notebook 的 **资源限额**、超时终止、**危险调用限制** 或只读根文件系统；**预装包白名单**。
- **存储**：用户目录与作业提交（对象存储 vs 磁盘）；**大文件**与防滥用。
- **可观测性**：执行错误、用量（计费与排障）。

## 6. LLM Chatbot 子系统

- **产品范围**：上下文（仅当前题 / 全章 / 全对话）；是否附带 **代码片段**；是否 **RAG**（向量库、切分策略、课程知识库接入）。
- **技术**：流式/非流、计费、**境内 endpoint** 可用性、限流、鉴权、**审计与提示注入防护**、费用与预算。

## 7. 工程与基础设施

- **代码与协作**：单仓/多仓；**CI**（单测、lint、Notebook 样例）；环境（dev/stage/prod）。
- **运行与部署**：云厂商与区域；K8s 或 PaaS；**HTTPS**、WAF、DDoS；**关系型库**、**缓存/队列**（异步批改、长任务）。
- **域名与备案**（若面向中国用户并接微信/QQ 等）：**ICP/公安备案**常是前置，列入时间与材料。
- **监控告警**：应用指标、Jupyter/容器健康、**LLM 错误率**、成本告警。

## 8. 合规、安全与隐私

- **隐私政策与用户协议**；**未成年人**条款如适用；**教育数据**存储与跨境。
- **内容安全与学术诚信**：题解与 AI 使用的课堂约定；**日志留存**策略。

## 9. 团队与项目治理

- **角色**：产品、课程编辑、后端/前端/DevOps、安全/合规、教学设计；无专人时在清单中标注**分期砍掉**或外包（例如二期再上 QQ 或自动判题）。
- **预算**：云资源、**LLM API**、Notebook 算力、证书、短信（若手机验证）等持续费用。

## 10. 建议先拍板的分叉（不先定会返工）

- **Notebook 技术路线**（JupyterHub 类 / 自研执行层 / 商业 Notebook API）。
- **微信/QQ 使用的产品形态**（网站应用与**备案域名**强相关）。
- **大模型**是否必须私有化或指定云、是否 RAG、**对话与代码**留存与可见范围。

# 续篇：分阶段、MVP、风险、待澄清

## 续篇 1. 建议的交付阶段（从准备到可运营）

| 阶段 | 目标 | 依赖准备（摘要） |
|------|------|------------------|
| M0 概念验证 | 浏览器内能跑通「登录 mock + 一个静态章 + 一个内嵌执行单元（哪怕是简化 REPL 或单细胞 Notebook）+ 调一次 LLM」 | 云账号、一个域名、模型 API 试用水位、Jupyter/替身的选型 spike |
| M1 教学端 MVP | 章节点播、**作业提交**（可先文件上传+存储）、学生列表级作业状态、**教师查看提交与简单打分** | 数据库模式、基础 RBAC、对象存储、最少监控 |
| M2 执行与体验 | **完整 Jupyter/Notebook 多租户**、资源限额、防滥用、作业与章节强绑定、站内通知 | 容器编排或托管 Notebook、安全评审、成本模型 |
| M3 身份与增长 | **账号密码 + 微信/QQ 登录**、备案与 OAuth 全链路、**隐私与条款**上站 | 备案完成、各开放平台应用审核、法律文本 |
| M4 智能辅导 | **Chatbot** 全站与题目上下文、可选 RAG 接课程知识、审计与防注入 | 向量库与分块、提示与护栏、对话留存策略 |

**分期原则**：「无本地环境、内嵌 Notebook、老师批改、OAuth」全部挤进一期会拉长上线并抬高合规成本；**常见做法**是 M1 先上教学+作业+账号密码+邮件，M3 再上微信/QQ 与**境内要求**强绑定的能力。

---

## 续篇 2. MVP 与「全量」的边界（避免范围泄漏）

- **MVP 可包含**：学生/教师角色、章与作业 CRUD、提交物存储、**教师打分+评语**、**嵌入一种受控的代码运行或 Notebook 路径（先一种即可）**、**最简 LLM 答疑（有免责声明与限流）**。
- **常后置**：QQ/微信二选一两端同时、RAG 全库、自动判题、直播、复杂学情分析。
- **与「无额外环境」强相关**：M0 就要验证**执行服务**的延迟与包管理策略（预装镜像 vs 动态安装），避免后期推翻 Notebook 方案。

---

## 续篇 3. 技术选型准备（清单之外的 spike）

- **Notebook**：在目标云上跑通 **JupyterHub / Binder 类**小集群，或商业 Notebook API 的 iframe 集成，记录：冷启动、并发热度、**单用户成本**。
- **LLM 网关**：流式/非流、计费、**境内 endpoint** 可用性、对话日志脱敏样例。
- **OAuth 回调**：在**已备案的测试子域**上跑通一种社交登录 sandbox。

---

## 续篇 4. 风险登记（准备期就要有人盯）

| 风险 | 说明 | 缓解 |
|------|------|------|
| 备案与 OAuth 时序 | 微信/QQ 网站应用要合法域名与备案材料 | 并行准备主体资质；开发期用密码登录 |
| 算力与 LLM 费用 | Notebook + 大模型**双高成本** | 限并发、包预装、模型分级（答疑用小模型试跑） |
| 安全与滥用 | 用户代码执行 = 高攻击面 | 网络隔离、超时、只读基镜像、全量审计 |
| 产品合规 | 教育数据、未成年人、AI 生成内容 | 条款、年龄门、老师可见范围设计 |

---

## 续篇 5. 待澄清问题（拍板后写正式设计规格与 writing-plans）

**下列每项有结论后，架构图与接口可定稿：**

1. **用户地域**：主战场是否**仅中国大陆**？（影响备案、支付、可接入的 LLM 与微信/QQ 产品形态。）
2. **Notebook 一期**：M1 是「内嵌**完整** Jupyter」还是 **「REPL/轻量 cells + 二期 Jupyter」**？
3. **微信与 QQ** 是否必须**同一发布火车**上齐？若只能先上一个，**优先微信还是 QQ**（与目标学生人群有关）？
4. **Chatbot**：练习页是否允许把**学生整段作业代码**发模型？**教师是否默认**可见对话？（强合规场景常设「可见」或「可开关」。）
