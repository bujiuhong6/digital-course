## 当前优先架构假设（讨论稿，见 §10 与续篇 5 可改）

下列结论来自近期方案收敛，**写入清单便于团队对齐**；若拍板变更，同步改 §10、续篇 5 与决策记录表。

- **部署**：**境外轻量云主机**对外提供 **HTTPS** 服务；**域名可在国内注册商购买**，DNS **A/AAAA 解析到境外 IP**（常见、与「网站托管在大陆机房需 ICP 备案」不是同一套流程，具体以注册商与监管要求为准）。
- **传输安全**：公网传递**学号、密码、作业与 Chatbot 内容**时，**HTTPS 视为必选项**（证书可用 Let’s Encrypt 等自动续期；成本主要是正确配置 443）。
- **学生端形态**：**Win / Mac 本地客户端**为主（作业可离线编辑、联网提交）；客户端技术栈候选见 **Electron / Tauri / Flutter 桌面** 等对比。
- **身份与汇聚**：学生用 **学号（及首登密码等）注册/登录**，提交物经网络汇总到**教师可见**的后台或教师端；**微信/QQ OAuth** 可作为**二期**能力，降低与「大陆网站备案+社交登录材料」绑定的首期压力。
- **大模型 / Jupyter 外网**：Chatbot 调公网 API、Notebook/执行环境若需 `pip` 或抓数据，需**受控出站**（白名单域名、预算与审计），与「整网裸开」区分开。

# 前篇：整体准备清单（十段）

## 1. 产品与教学设计（非代码）

- **用户与场景**：学生 / 教师 / 管理员；典型课节长度、同时在线人数、是否直播、是否线下班级绑定。
- **课程结构模型**：章 → 节 → 知识点 → 例习题；**作业与章节的对应**（一章多份、死线、重做次数、是否防抄袭）。
- **账号标识（与客户端方案）**：若以**学号**为唯一键，需规定**学号格式、重名/转班**处理；**桌面客户端**路径下，身份数据**本地缓存 + 与服务端同步**策略要产品化（冲突、离线提交队列）。
- **Jupyter 在教学中的角色**：全站统一笔记本还是每章一份模板；是否允许 `pip install`、是否开放网络、**单会话时长与 CPU/内存上限**（由平台在容器/Hub 配置或云规格中**显式设定**，非学生自选）。
- **评价与成绩**：计分规则（仅教师打分 / 自动测试占比）、成绩单导出、**人工批改界面**的字段（评语、按题给分、迟交扣减等）。

## 2. 「零本地环境」与「本地客户端」两条形态（尽早二选或分阶段）

**A. 纯浏览器 + 云端执行（原思路）**  
- **应用形态**：纯浏览器；**Python 在服务端或托管环境**执行。需明确 **Jupyter 嵌入方式**（JupyterHub、托管 Notebook、自研 jupyter-server + 前端 Thebe/iframe 等；各方案对多租户与运维要求不同）。  
- **多租户隔离**：一用户一容器 / 一会话一环境 / 共享池；**文件持久化**与作业草稿是否跨设备。

**B. 桌面客户端 + 境外 API 汇聚（当前优先讨论稿）**  
- **学生端**：本机安装 **Win/Mac 应用**，本地写作业；**联网**时向教师侧服务 **HTTPS 提交**（学号维度区分作业）。  
- **教师端**：至少具备**查看提交列表、按学号筛选、批改/打分**（浏览器管理后台或教师桌面客户端均可）。  
- **本机练习环境**：可内嵌轻量内核、**JupyterLite / Pyodide**、或本机起子进程（包体与合规各有利弊）；与 **A** 可**分阶段**：先满足「提交+汇聚」，再强化「全功能 Notebook」。

**共用：大模型数据流**  
- 题面与代码是否回传模型；对话是否持久化、是否对教师开放督导；**数据存储区域**（境外服务器时与**教育数据出境、校务规定**一并评估）。

**典型逻辑分层**（随形态替换组件名）：浏览器或桌面壳 → **认证 API** → 业务 API → 数据存储；可选：执行服务 / Jupyter；**LLM 网关**；提交物与日志落库。

## 3. 账号与权限（含微信 / QQ，可二期）

- **身份模型**：`User`、`Role`（学生/教师/管理员）、**班级/课程成员**、作业可见范围；**学号**作为学生主键时的**唯一性约束**与展示名规则。
- **凭据类**：密码策略、邮箱或手机验证（若做）、**找回密码**、会话方式（Cookie/JWT / 桌面 token）、是否上双因素。
- **微信 / QQ 登录**（可选、常二期）：各开放平台的应用类型；若未来切**大陆托管+网站**，**备案域名**、回调白名单、**主体与审核**材料；首期可用**学号+密码**降低耦合。
- **统一身份**：内部主键 + `oauth` 绑定表，避免多登录源冲突。
- **教师侧**：是否人工审核教师账号、邀请码、或后续与学校 SSO 对接。

## 4. 教学内容与作业子系统

- **内容管理**：富文本或 Markdown、代码块、附件；**版本与发布**（草稿/发布）；**编辑权限**（教师）。
- **作业系统**：题型（选择、简答、**Notebook 作业包**、自动测试）；**提交物**（`.ipynb`、日志、自动评分结果）；**批改队列**与通知（站内信/邮件/企业微信等，可选）；**桌面路径**下需定义**离线草稿与提交失败重试**。

## 5. Jupyter 嵌入与运行安全

- **运行时策略**：每 Notebook 的 **资源限额**、超时终止、**危险调用限制** 或只读根文件系统；**预装包白名单**。
- **存储**：用户目录与作业提交（对象存储 vs 磁盘）；**大文件**与防滥用。
- **可观测性**：执行错误、用量（计费与排障）。

## 6. LLM Chatbot 子系统

- **产品范围**：上下文（仅当前题 / 全章 / 全对话）；是否附带 **代码片段**；是否 **RAG**（向量库、切分策略、课程知识库接入）。
- **技术**：流式/非流、计费、**API 区域与可用性**（境外服务器调境外/境内模型各有限制）、限流、鉴权、**审计与提示注入防护**、费用与预算。

## 7. 工程与基础设施

- **代码与协作**：单仓/多仓；**CI**（单测、lint、Notebook 样例）；环境（dev/stage/prod）；**桌面**另加各平台**构建与签名**（Win/Mac 分发渠道）。
- **运行与部署**：**境外轻量云**与区域选型；单体或轻量 **K8s**；**HTTPS**（Let’s Encrypt 等）、WAF、DDoS；**关系型库**、**缓存/队列**（异步批改、长任务）。
- **域名**：**国内注册商购买**、解析到**境外主机**的常见组合；**备案**义务与「**服务器是否在大陆机房、是否对公众提供需备案类服务**」强相关，**以主管单位与云厂商说明为准**；架构讨论时区分**域名注册地**与**主机所在地**。
- **监控告警**：应用指标、Jupyter/容器健康（若启用）、**LLM 错误率**、成本告警、**客户端崩溃/版本**（若桌面端）。

## 8. 合规、安全与隐私

- **隐私政策与用户协议**；**未成年人**条款如适用；**教育数据**存储、**跨境**（境外云 + 国内师生）与校务要求。
- **内容安全与学术诚信**：题解与 AI 使用的课堂约定；**日志留存**策略。

## 9. 团队与项目治理

- **角色**：产品、课程编辑、后端/前端/DevOps、**桌面端**（若选 B）、安全/合规、教学设计；无专人时在清单中标注**分期砍掉**或外包。
- **预算**：云资源、**LLM API**、Notebook 算力、证书、短信（若手机验证）、**代码签名与分发**（若桌面）等持续费用。

## 10. 建议先拍板的分叉（不先定会返工）

- **产品主形态**：**纯 Web（A）** / **桌面客户端 + API（B）** / **分期先 B 后补 A 的浏览器教学**。
- **Notebook 技术路线**（JupyterHub 类 / 自研执行层 / 商业 Notebook API / **先 Pyodide 再升级**）。
- **微信/QQ**：首期做还是二期；若做，**大陆网站备案与 OAuth 材料**时间表。
- **大模型**：是否必须私有化或指定云、是否 RAG、**对话与代码**留存与可见范围。
- **域名与主机**：**境外主机 + 国内购域**是否定稿；是否另需**仅内网/学校**部署的灾备或切换方案。

# 续篇：分阶段、MVP、风险、待澄清

## 续篇 1. 建议的交付阶段（从准备到可运营）

| 阶段 | 目标 | 依赖准备（摘要） |
|------|------|------------------|
| M0 概念验证 | **桌面或 Web 二选一 spike**：学号登录 mock + **HTTPS** 通连境外 **API** + 一次作业提交落库 + **一次 LLM 调用**；若走桌面，验证 **Win/Mac 打样包** | 境外云账号、域名解析、TLS、模型 API 试用水位、客户端技术候选验证 |
| M1 教学端 MVP | **章内容 + 作业提交 + 教师按学号查看/打分**；提交以文件/JSON 为主可接受 | 数据库模式、基础 RBAC、对象存储或磁盘、最少监控 |
| M2 执行与体验 | **完整 Jupyter/Notebook 或多细胞执行**、资源限额、防滥用、作业与章节强绑定、站内通知 | 容器编排或托管 Notebook、安全评审、成本模型 |
| M3 身份与增长 | **账号密码已稳**后，可选 **微信/QQ**、**隐私与条款**上站与备案材料（若切大陆 Web 或社交登录要求） | 备案/开放平台时间表、法律文本 |
| M4 智能辅导 | **Chatbot** 全站与题目上下文、可选 RAG 接课程知识、审计与防注入 | 向量库与分块、提示与护栏、对话留存策略 |

**分期原则**：**境外 API + 学号密码**可先跑通闭环；**微信/QQ、大陆 ICP 流程、完整 Jupyter** 常后置，避免首期过度扩张。

---

## 续篇 2. MVP 与「全量」的边界（避免范围泄漏）

- **MVP 可包含**：学生/教师角色、**学号注册/登录**、章与作业 CRUD、提交物存储、**教师打分+评语**、**桌面或 Web 一条链跑通**、**最简 LLM 答疑（免责声明与限流）**。
- **常后置**：QQ/微信同时上线、RAG 全库、自动判题、直播、复杂学情分析、**全功能多租户 Jupyter**。
- **客户端路径**：先**提交汇聚 + 批改**再补**本机重型执行环境**，可降低首包体积与运维面。

---

## 续篇 3. 技术选型准备（清单之外的 spike）

- **客户端**：**Electron / Tauri / Flutter 桌面** 各打一个 **Hello + HTTPS 调登录 API** 包，评估体积、构建链、团队技能。
- **Notebook / 执行**：在目标环境跑通 **JupyterHub** 或 **JupyterLite 嵌入**，记录冷启动、成本、安全组。
- **LLM 网关**：流式/非流、计费、日志脱敏；**境外服务器访问所选模型 API** 的网络与账号。
- **OAuth（若二期）**：在**已定域名**上跑通回调（**大陆备案要求随托管位置变化**）。

---

## 续篇 4. 风险登记（准备期就要有人盯）

| 风险 | 说明 | 缓解 |
|------|------|------|
| 备案与 OAuth 时序 | 社交登录、**大陆托管网站**常与**备案、主体**绑定 | **首期学号+密码 + 境外 API**；二期再并行材料 |
| 算力与 LLM 费用 | Notebook + 大模型**双高成本** | 限并发、包预装、模型分级（答疑用小模型试跑） |
| 安全与滥用 | 用户代码执行 = 高攻击面；**HTTPS 与鉴权**失败会导致作业与账号风险 | 最小暴露面、限流、审计、**仅 TLS** 对外服务 |
| 数据跨境与校务 | **境外服务器**存学号、作业、对话 | 隐私条款、校务确认、区域/加密与保留策略 |
| 桌面分发与更新 | Win/Mac **签名、更新通道** | 早期定渠道与预算 |

## 续篇 5. 待澄清问题（拍板后写正式设计规格与 writing-plans）

**下列每项有结论后，架构图与接口可定稿：**

1. **用户地域**：主战场是否**仅中国大陆**？（影响模型选型、**数据出境**叙述、是否必须境内能力。）
2. **首期形态**：**以桌面客户端为主**还是**纯浏览器为主**？是否接受 **MVP 仅 HTTPS + 学号提交**、Jupyter **二期**？
3. **Notebook 一期**：M1 是「内嵌**完整** Jupyter」还是 **「轻量 cells / Pyodide + 二期」**？
4. **微信与 QQ**：首期是否必须？若二期，**优先**哪个与**大陆网站/备案**是否联动？
5. **Chatbot**：练习页是否允许把**学生整段作业代码**发模型？**教师是否默认**可见对话？
6. **域名与主机**：是否确认 **国内购域 + 境外轻量云 + 全站 HTTPS**？是否另需**纯内网灾备**？
7. **学号体系**：是否**全校唯一**、与学校教务系统**是否对账**（仅产品内生成 vs 导入名单）？
