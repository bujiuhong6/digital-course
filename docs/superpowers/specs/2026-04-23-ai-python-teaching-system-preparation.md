## 当前优先架构假设（已定稿范围，变更时同步 §10、续篇 5 与决策记录表）

- **部署**：**境外轻量云主机**对外提供 **HTTPS**；**域名可在国内注册商购买**，DNS **A/AAAA 解析到境外 IP**（与「网站托管在大陆机房需 ICP 备案」不是同一套流程，具体以注册商与监管要求为准）。
- **客户端形态**：仅 **Win / Mac 桌面客户端** + 后端 **HTTPS API**；**不包含**「纯浏览器 + 云端 Python 执行」作为并列产品方案（教学执行以**客户端内嵌**为主，见 §5）。
- **角色**：仅 **学生** 与 **教师**；**教师账号兼管**用户与内容（无独立管理员角色）。
- **作业与课程**：**一章一份作业**；**全班同一题目**；**无限次重做**；**不做防抄袭**；**无转班**业务。
- **身份**：学生以 **学号** 唯一注册/登录；**找回密码不做邮箱/手机验证**；**教师可查看全班学生密码**，并具备**重置任意学生密码**能力（按校务安全要求控制教师端访问与审计）。
- **LLM Chatbot**：采用**尽可能低成本的接入**——优先评估**国内大模型 API 的免费/试用额度**（如阿里云通义、智谱 BigModel、百度文心等**以各平台当时官网规则为准**）；**不做 RAG**，对话以**当前练习上下文 + 可选系统提示**为主。
- **Notebook / 代码执行**：选取**实现与运维负担最小**的路线；教学量与代码规模**不大**时，优先 **JupyterLite / Pyodide** 或**本机嵌入式 Python 子进程 + 极简沙箱**，避免一期上 **JupyterHub 集群**。

# 前篇：整体准备清单（十段）

## 1. 产品与教学设计（非代码）

- **用户与场景**：**学生、教师**两角色；**教师**负责建课、发作业、批改、密码与学生管理。
- **课程结构模型**：**章**；每章有 **教学内容** + **该章唯一一份作业**；全班题目相同；学生可**无限次重交**，保留**版本或仅保留最新**在规格中二选一；**不实现转班、不实现防抄袭**。
- **账号标识（桌面端）**：**学号** 全局唯一；**桌面端** 本地草稿与**联网提交**、失败重试策略。
- **练习与执行**：**Python 量不大**、课程不长 → **优先最轻量嵌入**（见 §5），由平台**设定**单会话/CPU/内存上限，学生不可自选。
- **评价与成绩**：以**教师给分+评语**为主；成绩单导出、迟交等规则在规格中**按需**收敛（本清单不强制防抄袭、多版本比对）。

## 2. 唯一产品形态：桌面客户端 + 境外 API

- **学生端**：本机 **Win/Mac 应用**；线下可编辑，联网 **HTTPS 提交**作业，按 **学号** 与 **章** 关联。
- **教师端**：**浏览器管理台** 或 **教师桌面端** 二选一在规格中定案；需支持：**章节内容、作业题面、按学号查看提交与历史重做**、**打分**、**查看/重置学生密码**。
- **大模型数据流**：题面与学生提问是否发往模型、对话是否**落库**、教师是否**审计对话**，在**隐私与校务**条款中写清；数据存**境外**时与**教育数据、出境**要求一并评估。

**典型逻辑分层**：桌面壳 → **TLS** → **鉴权与业务 API** → 关系型/对象存储；可选 **JupyterLite/本地内核**；**Chatbot** 经**国内 API 网关**（见 §6）；作业与元数据落库。

## 3. 账号与权限（无微信/QQ、无独立管理员）

- **身份模型**：`Role` ∈ {`student`, `teacher`}；**教师**拥有**用户与作业管理、密码查看与重置、内容发布**等「管理」能力，**不单独建 admin 表** 亦可（权限绑定 `teacher`）。
- **凭据类**：**学号+密码**；**无**邮箱/手机作为找回通道；**教师可见全班密码明文或可控展示** 与 **教师重置学生密码** 为产品必选项，实施时需**强审计、HTTPS、教师账号强保护**（否则法律与声誉风险高）。
- **不使用** OAuth 与微信/QQ 登录。

## 4. 教学内容与作业子系统

- **内容管理**：章级 **Markdown/富文本**、代码块与附件；**仅教师**可编辑发布。
- **作业系统**：**每章一题、全班同题**；**无限重做**；提交物可为 **`.ipynb` / 文本 / 打包**（规格中定一种或两种）；**批改队列**与通知**可选、轻量化**；桌面端**离线队列**定义清楚。

## 5. Notebook / 代码执行（最简优先）

- **目标**：在**小体量教学**下，**最快可用、最少运维**。
- **优先顺序（建议）**  
  1. **JupyterLite / Pyodide** 跑在**客户端内嵌 WebView**（无服务端多人内核、适合基础练习）。  
  2. 若必须 **CPython 库**：**本机打包的解释器子进程** + 受限工作目录 + 资源上限（仍比 **JupyterHub** 易落地）。  
  3. **JupyterHub / 一机多容器** 仅当 1/2 不足时再评估。
- **安全**：`pip install` 与**外网**是否开放为**显式产品开关**；**超时 kill**、**资源上限** 必配。

## 6. LLM Chatbot 子系统（免费优先、无 RAG）

- **产品范围**：**不建向量库、不做 RAG**；上下文以**当前题/当前章节 + 用户当轮输入** 为主；是否附带**代码片段** 在 UI 中可选。
- **「免费」落地方式**：在**国内**各开放平台注册，使用其**新客/限免/试用 token** 档（**通义、智谱、文心** 等，**具体额度与模型名以申请时官网为准**）；超出后按量计费或限流/降级/暂停，在预算中预留。
- **技术**：**HTTPS 出站** 至所选 API 域名；**流式/非流**、**限流**、**关键审计日志**、**防注入**；费用监控。

## 7. 工程与基础设施

- **代码与协作**：单仓优先；**CI**；**Win/Mac 构建与** **代码签名** 列入计划。
- **运行与部署**：**境外轻量云**、**Let’s Encrypt** 或云证书、**WAF/安全组** 仅开必要口；**关系型库** 存用户、章、作业、提交与**可选**对话摘要。
- **域名**：国内购域 + 解析境外；**备案** 与**大陆托管** 关系见假设节。
- **监控告警**：**API 错误、LLM 429/欠费、客户端崩溃率**（若可采集）。

## 8. 合规、安全与隐私

- **隐私与校务**；**未成年人** 如适用；**教育数据** 与**境外机** 存储的告知与同意。
- **教师可见学生密码** 属于**高敏感**设计，需**书面校务/团队共识** 与**访问留痕**；**内容安全** 与课堂 AI 使用规则可简化为**教师级说明**。

## 9. 团队与项目治理

- **角色**：产品、教学、**客户端**、后端、**运维** 可一人多职；**无**单独「OAuth 与备案专项」在首期。
- **预算**：**境外主机**、域名、**LLM 超额费用** 缓冲、**代码签名**；**大集群 Notebook** 非首期必选项。

## 10. 仍可在规格层微调的分叉

- **教师端** 用 **纯 Web 管理台** 还是 **双端桌面**（学生必装、教师可选浏览器）。
- **JupyterLite** 与 **本机 CPython 子进程** 的**分界**（依赖哪些 Python 包）。
- **学号** 是否由**教师手工导入** 或 **学生自注册后教师审核**（无转班前提下仍要防错号）。

# 续篇：分阶段、MVP、风险、待澄清

## 续篇 1. 建议的交付阶段（从准备到可运营）

| 阶段 | 目标 | 依赖准备（摘要） |
|------|------|------------------|
| M0 概念验证 | 学号登录 + **HTTPS** 通 API + 一章一作业 **提交/列表** + **一次** Chatbot 调通 + **JupyterLite 或**最简内嵌跑通 | 境外云、域名、TLS、**国内某家 API 密钥**、桌面 Hello 包 |
| M1 教学端 MVP | **多章**、每章一作业、**无限重做**、**教师打分**、**看/设密码/重置** | 库表、教师权限、**审计**字段 |
| M2 体验加固 | 客户端更新、**限额与超时** 打磨、**LLM 欠费/限流** 策略、**日志与备份** | 运营手册 |
| M3 可选加强 | 若确有需要再上 **本机 CPython 子进程** 或**更强沙箱**；**不做 RAG** 前提下不扩向量栈 |

**分期原则**：**少角色、同题、无 OAuth** 已省掉大量并行依赖；**别在一期**堆 **JupyterHub**。

---

## 续篇 2. MVP 与「全量」的边界

- **MVP 必含**：两角色、**一章一作业、同题、无限重做**、**教师看密码/重置**、**HTTPS**、**最简执行 + 免费/试用档 Chatbot**、**无 RAG**。
- **刻意不做**：**微信/QQ**、**防抄袭**、**转班**、**多管理员**、**RAG**、**高可用 Notebook 云**（除非 M3 真需要）。

---

## 续篇 3. 技术选型 spike（轻量）

- 桌面 **Electron 或 Tauri** 与后端 **通 HTTPS** 一条链。
- **JupyterLite** 嵌入一屏，跑通 3~5 个教学单元用例。
- 选定 **1 家国内 API**，读清**免费额度、QPS、可调用模型**；写死 **fallback** 文案（欠费/失败）。

---

## 续篇 4. 风险登记

| 风险 | 说明 | 缓解 |
|------|------|------|
| 教师通览密码 | 账号被盗则**全班**风险 | 教师强密码、**二因素**（仅教师可开）、**操作审计**、**HTTPS** 全站 |
| 免费 API 不可持续 | 额度用尽或**政策变价** | 监控用量、**预算上限**、**学生侧降级提示** |
| 学生代码执行 | 本机/嵌入解释器**滥用**与文件访问 | 沙箱、**工作目录**、**禁用危险模块** 列表、超时 |
| 数据在境外机 | 学号、作业、对话**出境** | 告知同意、**校务**确认、**最小**留存 |

---

## 续篇 5. 待澄清问题（写入 design 前）

1. **学生密码的存储形态**：**可逆**（为「教师可见」服务）** vs** **不可逆哈希 + 仅重置**？若必须「看见明文」，**加密可逆** 与**密钥**谁保管在规格中写死。
2. **重做**：保留**每次提交** 还是 **只保留最新**？影响存储与教师批改习惯。
3. **教师管理台** 必须 **Web** 还是 **可纯桌面**？
4. **国内 API** 三选一**首发** 哪一家？（在 spike 后写到决策表。）
